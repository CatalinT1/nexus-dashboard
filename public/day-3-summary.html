🔧 Changes Made

1

Header "New Appointment" now opens the create dialog

header.tsx

Previously, clicking New → New Appointment in the top-right header dropdown navigated to /appointments but the create dialog stayed closed — the user had to then click "New Appointment" again on the page itself.

Fixed by changing the link to /appointments?new=1. The appointments page detects this parameter and opens the create dialog immediately on load.

2

Auto-open create dialog via URL param ?new=1

appointments/page.tsx

Added useSearchParams and useRouter to the appointments page. A useEffect watches for ?new=1 in the URL — when detected, it calls openCreateDialog() (which also lazy-loads the client list) and then immediately replaces the URL back to /appointments to keep the address bar clean.

  • A useRef guard prevents the dialog from opening twice on re-renders
  • router.replace() is used (not push()) so the back button doesn't loop back to the param URL
  • Works from any navigation source — header dropdown, direct link, or future deep links
3

Inline new client creation from appointment dialog

appointments/page.tsx

When the user opens the New Appointment (or Edit Appointment) dialog and the client they need doesn't exist yet, they no longer have to close the dialog, go to Clients, create the client, and come back.

  • The Client dropdown now has a ＋ New client… option at the bottom (indigo text, separated by a line)
  • Selecting it opens a small dialog: Full Name (required), Email (required), Phone (optional)
  • Clicking Create & Select inserts the client into Supabase, adds them to the client list, and automatically selects them in the appointment form
  • The mini-dialog closes and the user is back in the appointment form with the client already filled in
  • The new client appears sorted alphabetically in the dropdown for all future use
  • Works from both the Create and Edit appointment dialogs

💡 Technical Notes

Why useRef instead of state for the auto-open guard?
A useState boolean would trigger a re-render when set, which could cause the useEffect to run again before the URL is replaced. A useRef is synchronously readable within the effect without causing re-renders — making the guard reliable even in React Strict Mode's double-invocation.
Why router.replace() and not router.push()?
push() adds a history entry — pressing Back would return to /appointments?new=1 and re-open the dialog. replace() swaps the current history entry so the back button behaves naturally.
