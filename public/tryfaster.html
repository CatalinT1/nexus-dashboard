🚀 Performance Fixes

1

Dashboard: Stop loading all historical appointments

Performance · High

The dashboard was fetching every appointment ever created (select("*") with no date filter) just to display 5 upcoming appointments and count today's total. On a busy clinic this would grow to thousands of rows over time — all transferred from the database on every dashboard load.

Fixed by adding a gte("start_time", todayStart) filter, excluding already-cancelled/completed appointments at the database level, and capping results at 20 rows. Also added a precise column projection instead of *, reducing payload size further.

Before — all rows, all time
supabase .from("appointments") .select("*") .order("start_time") // → fetches ALL historical rows // Every appointment ever created // No limit · No date filter
After — scoped, projected
supabase .from("appointments") .select("id,client_name,title,...") .gte("start_time", todayStart) .not("status","in",'("cancelled","completed")') .order("start_time") .limit(20) // Only today + future · max 20 rows // Only the 16 columns actually used

Impact: For a clinic with 2 years of history this could drop from ~500+ rows to ~5–20 rows per dashboard load.

2

Dashboard: 4 client COUNT queries → 1 status query

Performance · High

The dashboard KPI section ran four separate COUNT queries against the clients table (total, active+VIP, VIP only, prospects only) — each one a separate round-trip to Supabase. That's 4 of the 8 total dashboard queries just for client counts.

Before — 4 round trips
// Query 1 supabase.from("clients") .select("*", { count: "exact", head: true }) // Query 2 supabase.from("clients") .select("*", { count: "exact", head: true }) .in("status", ["active","vip"]) // Query 3 + Query 4 (same pattern)
After — 1 round trip
// One query, one column supabase.from("clients") .select("status") // Count client-side (zero extra queries) totalClients = statuses.length activeClients = statuses .filter(c => c.status === "active" || c.status === "vip").length // vipClients, prospectClients same way

Impact: Reduced dashboard from 8 parallel queries to 5. Saves 3 database round-trips on every single dashboard page load.

3

Header: One Supabase client instance, not one per operation

Performance · Medium

The Header component was calling createClient() inside every useEffect and every event handler — notifications fetch, search debounce, mark-read, mark-all-read, and sign-out. That's up to 5 separate client constructions per header mount (every page navigation).

Before — new client per op
// In notification useEffect const supabase = createClient(); supabase.from("notifications")... // In search debounce const supabase = createClient(); await Promise.all([...]) // In handleMarkRead const supabase = createClient(); await supabase...
After — shared instance
// Once at component scope const supabase = useMemo( () => createClient(), [] ); // All handlers share it useEffect(() => { supabase.from("notifications")... }, []); const handleMarkRead = async (id) => { await supabase.from(...)... }
4

Clients + Appointments pages: shared Supabase client via useRef

Performance · Medium

Both the Clients and Appointments pages were creating a new createClient() instance inside every handler (delete, edit, status-change, create, etc.). This means every button click constructs a brand new Supabase client object. Fixed with useRef(createClient()).current — created once at mount, stable across re-renders and closures.

Before — inside every handler
const handleDeleteClient = async () => { const supabase = createClient(); await supabase.from(...)... } const handleEditClient = async () => { const supabase = createClient(); await supabase.from(...)... } // × 6+ handlers in appointments page
After — stable ref
// Once, at component mount const supabase = useRef(createClient()).current; const handleDeleteClient = async () => { await supabase.from(...)... } const handleEditClient = async () => { await supabase.from(...)... }
5

Dashboard loading skeleton — instant visual feedback

UX · Medium

Added src/app/(dashboard)/dashboard/loading.tsx. Next.js App Router automatically shows this file while the server-side dashboard page is fetching data. Previously users saw a completely blank screen until all 5 Supabase queries completed.

The skeleton mirrors the exact layout of the real dashboard — KPI cards, chart area, bottom row, quick stats — using animate-pulse shimmer effect. Perceived load time drops from "blank" to "instant" even if actual data fetch time is unchanged.

🗄️ Database Indexes Added (17)

Indexes were applied via Supabase MCP apply_migration. They speed up the WHERE/ORDER BY clauses used throughout the app. Without indexes, Postgres does a full table scan for every query — with indexes it jumps directly to the matching rows.
idx_appointments_start_time
Speeds up date-range filters (gte start_time) used in dashboard and appointments page.
idx_appointments_status
Speeds up NOT IN (cancelled, completed) filter on upcoming appointments.
idx_appointments_client_id
Speeds up client profile page: "all appointments for this client."
idx_appointments_org_status
Composite: org + status. Covers the most common RLS-filtered + status-filtered query pattern.
idx_appointments_org_start
Composite: org + start_time. Covers the dashboard upcoming-appointments query under RLS.
idx_clients_status
Speeds up status filter on clients list and count queries.
idx_clients_total_revenue
Speeds up Reports "top clients by revenue" ORDER BY query.
idx_clients_org_status
Composite: org + status. Covers org-scoped dashboard client-count query.
idx_notifications_is_read
Speeds up unread-count lookups for the bell badge in the header.
idx_notifications_created_at
Speeds up ORDER BY created_at DESC used in header bell and /notifications page.
idx_notifications_org_read
Composite: org + is_read + created_at. Covers the most frequent org-scoped unread query.
idx_client_activities_client_id
Speeds up client profile page activity feed (eq("client_id", id)).
idx_client_activities_created
Speeds up dashboard recent-activity feed (ORDER BY created_at DESC LIMIT 5).
idx_staff_is_active
Speeds up dashboard active-staff count (eq("is_active", true)).
idx_revenue_data_month_order
Speeds up revenue charts (ORDER BY month_order ASC) used on dashboard and reports.
idx_clients_total_revenue (DESC)
Covers the descending sort on Reports "top clients" tab without a full table sort.
idx_appointments_org_* (composite)
All org-scoped queries benefit most from composites — Postgres can skip the org filter and jump straight to the data.

🐛 Bug Fixes

Settings "Dark Mode" switch does nothing Broken feature settings/page.tsx The switch was a bare <Switch /> with no checked or onCheckedChange props. Now wired to useTheme(): checked={theme === "dark"} onCheckedChange={(v) => setTheme(v ? "dark" : "light")}.
Missing organization_id on INSERTs (from Day 4) Data error clients/new, appointments, staff Creating new clients, appointments, or staff members would fail with a NOT NULL constraint error after the multi-tenancy migration. Fixed in 4 INSERT sites by adding organization_id: orgId via getOrgId().
Reports period selector does nothing Non-functional UI report-charts.tsx The Month / Quarter / Year dropdown sets period state but that state is never used to filter the charts or data. The data always shows the full year regardless of selection. Note: fixing this fully requires server-side date filtering — flagged as future work.
All KPI trend percentages are hardcoded Data accuracy dashboard/page.tsx, report-charts.tsx The "+18.2%", "+12.5%", "+8.3%" values shown on KPI cards are static strings, not computed from comparing current vs previous periods. They will always show the same values regardless of actual business data. Flagged: requires period-comparison queries.
Reports category breakdown is fake data Misleading report-charts.tsx The "By Category" pie chart uses a hardcoded categoryData array (Consultation 38%, Follow-up 28%, etc.) that never reads from the database. Flagged: requires a GROUP BY category query.
