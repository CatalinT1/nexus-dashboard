🔧 Implementation Phases

1

Type Fix — AppointmentCategory

src/lib/types.ts

The TypeScript type had "emergency" as a category option, but the Supabase database enum only includes "consultation" | "follow-up" | "treatment" | "review" | "other". This caused silent failures on insert/update. Fixed the type and all dropdown options to use "other".

2

Staff — Edit, Delete, Toggle

staff/page.tsx

Complete staff management. StaffCard now accepts onEdit, onDelete, onToggleActive, and toggling props. All three operations call Supabase and update local state via mapStaff().

  • Edit dialog (max-w-lg): name, email, phone, role (Select), specialization, 7 interactive day-toggle buttons (Sun–Sat, indigo when active), start time + end time inputs
  • Delete dialog (max-w-sm): confirmation with the staff member's name. If the staff has appointments, Supabase returns a FK constraint error (ON DELETE RESTRICT) — this error is displayed inside the dialog rather than crashing
  • Toggle: flips is_active, updates card status dot (green ↔ grey) instantly
  • Work day toggle pattern: clicking a day adds/removes it from workDays: number[] array (0=Sun … 6=Sat)

Client — Delete

clients/page.tsx

The "Delete" item in the client row dropdown was previously a no-op. Wired it to a confirmation dialog with deleteTarget state and a handleDeleteClient() handler that deletes from Supabase and filters the row out of the local list.

Appointments — Status Actions (Confirm / Cancel)

appointments/page.tsx

The Confirm and Cancel buttons in the appointment detail dialog were rendered but had no handlers. Added handleStatusChange(id, status) which calls supabase.from("appointments").update({ status }), then splices the result back into both the appointments array and the selectedApt state so the badge updates instantly.

Appointments — Create Dialog

appointments/page.tsx

"New Appointment" button now opens a full max-w-2xl form dialog with sections for appointment details, client & staff, schedule, and additional options.

  • Lazy client loading: clients are not fetched on page mount — only when the dialog first opens, with a clientsLoaded guard to prevent duplicate fetches
  • Client Select: dropdown populated from clients table (id, name, email)
  • Staff Select: filtered to active staff only, shows specialization as secondary label
  • Category: consultation, follow-up, treatment, review, other
  • Color picker: 8 preset swatches — indigo, cyan, emerald, violet, pink, orange, amber, slate
  • Fee: numeric input (step 0.01), Location: text, Notes: textarea
  • Reminders: Switch toggle
  • Denormalization: on insert, client_name and staff_name are looked up from state and written alongside the FK IDs (required by the DB schema)

Appointments — Edit & Delete

appointments/page.tsx

Edit and delete actions added to the detail dialog footer. The edit flow required special handling because the detail dialog closes before the edit dialog opens — editTargetId state holds the appointment ID across this transition, while selectedApt is cleared.

  • Edit: pre-populates date as ISO slice(0,10) and time as toTimeString().slice(0,5), same form fields as create
  • Delete: red button in detail footer opens a max-w-sm confirmation, then removes from the calendar and list view
7

Users Management Page + API Routes

users/page.tsx · api/admin/

A brand-new /users page for admins to manage all dashboard user accounts. Added to the sidebar between Staff and Reports.

  • Table: avatar, name, email, role (editable Select), status badge (Active/Inactive), joined date, actions dropdown
  • Role change: inline Select dropdown per row → calls PUT /api/admin/update-profile → optimistic update in state
  • Activate/Deactivate: dropdown toggle → same API route
  • Self-protection: current user's row shows role as a Badge (not editable) and has no dropdown — prevents accidental self-demotion
  • Invite dialog: name, email, role → POST /api/admin/invite-user → Supabase sends native invite email; profile created when they accept
  • Service Role Key: both API routes use createClient(URL, SUPABASE_SERVICE_ROLE_KEY) to bypass RLS — required because the RLS policy only allows users to update their own profile row

💡 Key Technical Decisions

DecisionWhy
Lazy client loading for appointment dialog Clients are not fetched when the appointments page loads — only when the create/edit dialog first opens. A clientsLoaded boolean prevents duplicate fetches on subsequent opens. This avoids an unnecessary network call for users who never open the dialog in a session.
Separate editTargetId state When editing an appointment, the detail dialog closes before the edit dialog opens. selectedApt is cleared by the detail dialog's onOpenChange. A separate editTargetId string state persists the ID through this transition without being tied to the dialog's open/close cycle.
FK error surfaced in delete dialog The appointments.staff_id column has ON DELETE RESTRICT. Rather than pre-checking or hiding the delete option, the Supabase error string is caught and displayed inside the delete dialog — simpler and more transparent to the user.
Service role key for admin ops The profiles table RLS policy "Users can update own profile" only allows self-update. An admin changing another user's role must bypass RLS. The service role client is used server-side only (API routes, never client components) and never exposed to the browser via a NEXT_PUBLIC_ prefix.
Self-protection on Users page The current user's row shows role as a read-only Badge and has no actions dropdown. This prevents an admin from accidentally demoting themselves or deactivating their own account — a scenario that would lock them out of the admin functions.
Denormalized client_name / staff_name The appointments table stores both the FK IDs and the human-readable names. On create/update, the names are looked up from the in-memory state and written alongside the IDs. This avoids expensive joins on every calendar render and matches the existing schema design from Day 1.

🔌 New API Routes

Server Route

PUT /api/admin/update-profile

Updates another user's role and/or is_active in the profiles table.

  • Auth guard: verifies caller is authenticated
  • Role guard: verifies caller's profile role is "admin"
  • Uses service role client to bypass RLS
  • Body: { targetId, role?, is_active? }
  • Returns: updated profile row
Server Route

POST /api/admin/invite-user

Sends a Supabase invite email to a new user. Supabase handles the email; when the user accepts, the handle_new_user() DB trigger auto-creates their profile row.

  • Auth guard + admin role check
  • Uses admin.auth.admin.inviteUserByEmail()
  • Body: { email, name, role }
  • Returns: { user: data.user }
  • Requires: SUPABASE_SERVICE_ROLE_KEY in .env.local
